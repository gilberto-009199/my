Introdução à Integração de Sistemas Utilizando Apache Camel

Pablo Oliveira

·

Published in

CWI Software

·

May 5, 2020

O Apache Camel é um framework de integração open source que permite integrar rápida e facilmente vários sistemas que consomem ou produzem dados.

Introdução

Atualmente, é difícil encontrar uma solução de software que funcione isoladamente, sem depender de nenhuma funcionalidade ou recurso externo. Desenvolver a integração entre esses sistemas pode ser uma atividade trabalhosa, pois as aplicações que devem ser interligadas, normalmente, utilizam protocolos, paradigmas e linguagens diferentes¹.

Gregor Hohpe e Bobby Woolf, em Enterprise Integration Patterns (EIP)² , apresentam padronizações para descrever, documentar e implementar problemas complexos de integração e os programadores podem se valer deste catálogo para idealizar suas soluções. Mas, embora este livro seja um excelente direcionador, ainda é necessário implementar o código¹.

Entende-se que implementar do zero toda a estrutura necessária para a comunicação entre os sistemas é uma estratégia muito cara e arriscada por conta da complexidade. Os desenvolvedores que atuam como integradores reconhecem esta dificuldade, assim como reconhecem que o foco principal deveria ser as regras de negócio.

Um bom framework de integração fornece uma interface simples e gerenciável que permite utilizá-lo de forma fácil, abstraindo a implementação dessa estrutura complexa de padrões e protocolos¹. O Apache Camel é um bom exemplo disto e neste artigo serão apresentados alguns dos seus conceitos e a sua aplicação em um exemplo prático.

1. Sobre o Apache Camel

Essencialmente, o Apache Camel é um roteador (routing engine), ou seja o Camel roteia os dados entre dois endpoints. Cabe ao programador definir a origem dos dados (from), o seu destino (to), e a forma com a qual ele será processado (mediation engine)¹. Um dos princípios fundamentais do Camel é que ele não determina o tipo de dado que precisa ser transacionado. Este alto nível de abstração facilita a interação com vários sistemas porque aproveita-se a mesma API, independentemente do protocolo ou tipo de dado utilizado.

Nesta seção, é interessante mencionar o que o Camel não é. O Camel não é um barramento de serviço corporativo (ESB), embora alguns o referenciem com um pequeno ESB por conta do seu suporte a roteamento, transformação, monitoramento, orquestração e outras características¹.

2. Utilizando a Camel DSL

A domain-specific language (DSL) do Camel suporta linguagens como Java, Scala, Groovy e também permite que regras de roteamento sejam especificadas em XML. O objetivo da DSL é permitir que o desenvolvedor se concentre no negócio e não na ferramenta de integração.

Rotas

No Camel, as rotas interligam os endpoints determinados pelos métodos from() e to(), sendo origem e o destino, respectivamente. Como parâmetros destes métodos, são fornecidas as URIs de acordo com o componente utilizado. Entre os dois endpoints, o programador pode definir transformações, filtros, divisões, validações e outras regras para tratamento dos dados. No exemplo prático da seção 3, será apresentado o uso de alguns recursos disponíveis na DSL.

3. Aplicação de Exemplo

A Figura 1 apresenta a visão geral de uma aplicação de exemplo que tem como objetivo utilizar o Apache Camel para ler um arquivo XML e, para cada nome de Pokémon que consta na lista, buscar suas informações na PokéAPI. Por fim, os dados consultados na PokéAPI serão gravados em uma simulação de Pokedéx que fornece uma interface de comunicação via API Rest.

Arquivo XML de Entrada

A entrada da rota programada é um arquivo XML simples com uma lista de nomes de Pokémons a serem gravados na Pokedéx simulada. Abaixo um exemplo da estrutura do arquivo:

<pokemons>
    <name>bulbasaur</name>
    <name>charmander</name>
</pokemons>

PokéAPI

A PokéAPI é uma API RESTful vinculada a um extenso banco de dados sobre a série Pokémon. Ela foi utilizada como fonte para obter as informações dos Pokémons mencionados no XML de entrada.

API Rest Pokedéx

Para este projeto de exemplo, foi criada uma API simples utilizando Spring Boot para receber via POST os dados do Pokémon a ser registrado. Por se tratar de uma situação para fins de exemplo de integração com APIs Rest utilizando Camel, não foi implementada nenhuma estrutura para armazenamento das informações. Ao receber uma requisição, a Controller apenas registra no console os dados recebidos.

Rota de Integração com Apache Camel

O bloco 3 apresenta a implementação completa da rota e a seguir serão detalhadas as etapas:

  • Etapa 1- Leitura do arquivo XML: a instrução from("file:input-pokemon?noop=true"), busca os arquivos do diretório input-pokemon no qual está o XML de exemplo. A seguir, utiliza-se o padrão de splitter e o padrão xpath para dividir o conteúdo do arquivo em diversas mensagens, uma para cada elemento <name>: split().xpath("/pokemons/name").
  • Etapa 2 - Consulta na PokéAPI: a instrução to("direct:rota-pokeapi") encaminha o fluxo para a rota from("direct:rota-pokeapi") que por sua vez inicia o processo de consulta do Pokémon na PokéAPI. A instrução setProperty("pokemonName", xpath("/name/text()")) cria uma propriedade com o nome do Pokémon a ser consultado. Os métodos setHeader configuram os cabeçalhos da requisição e, finalmente, o método to() envia a mensagem para o endpoint da PokéAPI.
  • Etapa 3 - Registro do Pokémon na Pokedéx: com a mensagem de retorno da instrução to("http4://pokeapi.co/api/v2/pokemon") temos no body da mensagem o json com os dados do Pokémon consultado. Tendo isto, utiliza-se novamente os métodos setHeader para configurar os cabeçalhos de uma nova requisição e, então, é acionada a API da Pokedéx por meio do to("http4://localhost:8080").

Aqui abaixo, está o log no console de execução do componente que implementa da rota com o Camel:

INFO 22:11:51.994 - Salvando dados do Pokemon na Pokedex
INFO 22:11:52.012 - Buscando dados do Pokemon: charmander

Este é um exemplo de log no console de execução da Pokedéx API:

[POST] - Pokemon{name='charmander', id=4, height=6, weight=85}

Fluxo da Rota

A rota inicia no método from() da linha 8 do Bloco 3, onde o estímulo inicial é um arquivo XML na pasta input-pokemon. Na sequência, o Camel cria uma Exchange. Conforme ilustrado na Figura 3, Exchange é o container da mensagem que será transportada ao longo da rota.

A rota-principal encaminha a Exchange do fluxo primeiramente para a sub-rota rota-pokeapi. Nesta primeira sub-rota, o body da In message contém o a tag <name>. Ao finalizar a transação com a PokeAPI, o body da In message contém o json com os dados do Pokémon consultado. Na continuação, a Exchange retorna para a rota-principal que, por conseguinte, a envia para a sub-rota final (rota-pokedex). A rota-pokedex faz uma requisição para a Pokedex API, utilizando o json do body na In message como payload.

Diante do que foi apresentado, constata-se que a Exchange parte da rota-principal e circula de forma síncrona entre as duas sub-rotas, ou seja o retorno do to("direct:rota-pokeapi") segue como entrada para o to("direct:rota-pokedex"). É possível encaminhar a Exchange de forma assíncrona para as duas sub-rotas ao mesmo tempo. Para isto, basta alterar o prefixo direct do método to() para o prefixo seda. Ficando da seguinte forma:

from("file:input-pokemon?noop=true"). 
...
to("seda:rota-pokeapi"). 
to("seda:rota-pokedex");

O importante nesta abordagem é lembrar que, desta forma, a Exchange de entrada será a mesma para ambas as sub-rotas. Como consequência, os dados de entrada também serão os mesmos. Compete ao desenvolvedor avaliar o mecanismo que atende a necessidade da demanda.

Reprodução do Exemplo

Os fontes da aplicação de exemplo estão disponíveis no Github, bem como o roteiro para execução. No repositório, estão disponíveis dois artefatos: um componente maven que sobe um Spring Boot para a API da Pokedéx e um componente maven simples com uma classe/método main que executa a Rota com o Camel. Inclusive, o Camel já possui suporte para uso com Spring, mas esta abordagem não está contemplada neste exemplo.

4. Considerações Finais

A partir dos estudos e aplicações práticas realizadas, confirma-se que o Camel é uma solução robusta e que entrega o que se propõe: ser um framework de integração que facilita o trabalho do desenvolvedor para que ele se concentre nas regras de negócio. A aprovação de uma grande comunidade usuária aliada ao suporte de uma sólida mantenedora como a Apache Software Foundation, garantem a segura do uso do Camel em soluções finais.

Este artigo abstraiu e generalizou uma série de conceitos relevantes sobre a arquitetura do Camel, focando em apresentar uma visão geral sobre o tema e uma aplicação prática da solução.
